PRIMARY HOOKS BREAKDOWN
================================================================================

1. SESSION-INIT.CJS (201 LOC)
   Purpose: Session initialization and environment setup
   Key Functions:
   - Load and merge project config (.ck.json + ~/.claude/.ck.json)
   - Detect project type (Node/Python/Go/Rust/Java)
   - Detect package manager (npm/pip/cargo/go)
   - Resolve active plan (explicit session or branch-matched)
   - Compute naming patterns ({date}-{issue}-{slug})
   - Initialize 40+ environment variables (CK_*)
   
   Output: ~300 tokens of session context + coding level guidelines

2. SUBAGENT-INIT.CJS (167 LOC)
   Purpose: Inject compact context (~200 tokens) to subagents
   Key Functions:
   - Compute naming patterns for reports and plans
   - Resolve absolute paths based on subagent CWD
   - Include plan info, paths, rules, agent-specific context
   - Support monorepo/subdirectory workflows (Issue #327)
   - Optional trust verification via passphrase
   
   Output: JSON structure with additionalContext field

3. DEV-RULES-REMINDER.CJS (52 LOC)
   Purpose: Inject development rules at each user message
   Key Functions:
   - Avoid duplicate injection (check transcript)
   - Build session context (status, rules, file size warnings)
   - Provide modularization hints for large files
   
   Output: ~200 tokens (conditional, not always injected)

4. SCOUT-BLOCK.CJS (117 LOC + SUBSYSTEM ~1,100 LOC)
   Purpose: Block access to heavy directories and overly broad patterns
   Key Functions:
   - Load .ckignore patterns (gitignore-spec)
   - Extract paths from tool input (file_path, command, pattern)
   - Detect broad patterns (**, **/*.ts, etc.)
   - Match paths against patterns
   - Format user-friendly error messages
   
   Subsystem Modules:
   - pattern-matcher.cjs: Gitignore-spec matching
   - path-extractor.cjs: Extract paths from tools
   - broad-pattern-detector.cjs: Detect context-filling patterns
   - error-formatter.cjs: Format error messages
   - vendor/ignore.cjs: Gitignore parser (~850 LOC)
   
   Exit Codes: 0=allowed, 2=blocked
   Default Blocked: node_modules, __pycache__, .git, dist, build, etc.

5. PRIVACY-BLOCK.CJS (145 LOC)
   Purpose: Prevent access to sensitive files without user approval
   Key Functions:
   - Detect sensitive patterns (.env, credentials, keys, etc.)
   - Allow safe patterns (.env.example, .env.sample)
   - Enforce APPROVED: prefix for user-approved files
   - Detect path traversal and suspicious paths
   - Format AskUserQuestion JSON markers for LLM integration
   
   Exit Codes: 0=allowed, 2=blocked
   Sensitive Patterns: .env, credentials, secrets, .pem, .key, id_rsa, id_ed25519
   Safe Patterns: .example, .sample, .template

6. NOTIFICATIONS (MULTI-PROVIDER SYSTEM)
   Purpose: Send session completion notifications to external services
   Key Functions:
   - Load environment with cascade (process.env > ~/.claude/.env > ./.claude/.env)
   - Route to enabled providers (Telegram, Discord, Slack)
   - Send formatted messages with event details
   - Throttle errors (5-minute quiet period)
   
   Providers:
   - telegram.cjs (109 LOC): Telegram Bot API with Markdown
   - discord.cjs (197 LOC): Discord Webhooks with Embeds
   - slack.cjs (111 LOC): Slack Block Kit format
   
   Trigger Events: Stop, SubagentStop, Notification, AskUserPrompt
   Features: Zero dependencies, native fetch, non-blocking (always exit 0)

================================================================================
SECURITY ARCHITECTURE (3 LAYERS)
================================================================================

LAYER 1: SCOUT BLOCKING
  Strategy: Gitignore-spec pattern matching with normalization
  Patterns: node_modules, __pycache__, .git, dist, build, .next, etc.
  Exceptions: Build commands (npm build, cargo build, docker build, etc.)
  Broad Pattern Detection: Blocks **, **/*.ts, *.ts at root
  Exit Code: 2 (blocked)

LAYER 2: PRIVACY ENFORCEMENT
  Strategy: Regex-based sensitive file detection with user approval
  Sensitive: .env*, credentials, secrets, .pem, .key, id_rsa, id_ed25519
  Safe: .env.example, .env.sample, .env.template
  Approval: APPROVED: prefix allows access after user confirmation
  Exit Code: 2 (blocked) or 0 (approved/allowed)

LAYER 3: BUILD COMMAND ALLOWLIST
  Allows: npm|pnpm|yarn|bun (run) build|test|lint|dev|start|install
  Allows: npx, tsc, vite, webpack, rollup, go, cargo, make, mvn, gradle
  Allows: docker, kubectl, terraform, and many other tools
  Venv Executables: .venv/bin/*, venv/Scripts/* (cross-platform)
  Exit Code: 0 (allowed)

================================================================================
DESIGN PATTERNS
================================================================================

1. FAIL-OPEN ARCHITECTURE
   - All hooks exit 0 on errors
   - Prevents denial of service
   - Errors logged but don't block

2. MODULAR REUSABILITY
   - Core logic in lib/ for reuse
   - Claude hooks, OpenCode plugins, future integrations
   - Clear separation of concerns

3. ENV VAR CASCADE
   - Priority: process.env > ~/.claude/.env > ./.claude/.env
   - Global defaults, project overrides, runtime overrides

4. JSON MARKERS FOR LLM INTEGRATION
   - Privacy block embeds PRIVACY_PROMPT JSON
   - LLM parses and calls AskUserQuestion
   - Seamless user approval workflow

5. SESSION STATE vs. BRANCH PLAN
   - Active Plan: Explicitly set, used for reports
   - Suggested Plan: Auto-detected from branch, for UI hints
   - Prevents stale plan pollution on fresh sessions

6. CWD-BASED PATH RESOLUTION
   - All paths absolute and relative to CWD
   - Git root tracked separately for reference
   - Enables monorepo and subdirectory support

KNOWN LIMITATIONS
================================================================================

1. Pattern Matching - Gitignore-spec limitations with complex shell patterns
2. Privacy Block - Regex-based, may miss edge cases
3. Broad Pattern Detection - Heuristic-based, not AST-aware
4. Notification Throttling - Global file state, not per-provider
5. Subdirectory Support - Requires explicit CWD from Claude Code
6. No Caching - Patterns reloaded per check (performance trade-off)
