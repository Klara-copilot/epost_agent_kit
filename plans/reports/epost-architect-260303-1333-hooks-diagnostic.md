**Date**: 2026-03-03

**Request**: Identify problems with all hooks that cause errors on start

## Important: Source of Truth

`.claude/` is **generated output** -- wiped and regenerated by `epost-kit init`. All fixes MUST go to `packages/core/` (the canonical source). Paths below list both the source file (where to fix) and the generated file (where it manifests).

## Summary

Found **4 problems** across the hook system. Two are **critical** (cause visible errors every session), two are **moderate** (latent bugs).

## Problems Found

### P1 [CRITICAL] PostToolUse `npm run lint` and `npm run build` fail every time

**Source**: `packages/core/settings.json` -- PostToolUse hooks (lines 50-70)
**Manifests**: `.claude/settings.json`
**Error**: `ENOENT: no such file or directory, open 'package.json'`
**Cause**: No `package.json` at project root. This is a multi-package monorepo.
**Impact**: Every Write/Edit tool call triggers two failing npm commands (lint + build). Each command takes up to 30s/60s timeout before `|| true` swallows the error. Slows down every file edit.

**Fix options**:
- A) Remove these PostToolUse hooks entirely (recommended -- no root package.json means these are meaningless)
- B) Change commands to `cd epost-agent-cli && npm run lint -- --fix` and `cd epost-agent-cli && npm run build`
- C) Add a guard: `test -f package.json && npm run lint -- --fix || true`

**Recommendation**: Option A -- remove them.

### P2 [CRITICAL] statusLine command references missing script

**Source**: `packages/core/settings.json` -- statusLine config (lines 112-116)
**Manifests**: `.claude/settings.json`
**Config**: `"command": "~/.claude/statusline.sh"`
**Error**: `~/.claude/statusline.sh` does not exist
**Impact**: Status line fails silently on every prompt. May cause visible error flicker depending on Claude Code version.

**Fix**: Either create the script or remove the `statusLine` block from `packages/core/settings.json`.

### P3 [MODERATE] session-init.cjs uses `os` without importing it

**Source**: `packages/core/hooks/session-init.cjs`, line 87
**Manifests**: `.claude/hooks/session-init.cjs`
**Code**: `process.env.USERNAME || process.env.USER || process.env.LOGNAME || os.userInfo().username`
**Cause**: `os` module is not imported in session-init.cjs (it IS imported in `lib/ck-config-utils.cjs` but that's a different module scope)
**Impact**: Currently masked because `process.env.USER` is always set on macOS. Would crash on systems where USER/USERNAME/LOGNAME are all unset (some Docker containers, CI).
**Fix**: Add `const os = require('os');` at top of `packages/core/hooks/session-init.cjs`, or remove the `os.userInfo()` fallback.

### P4 [MODERATE] privacy-block.cjs uses async stdin read while scout-block.cjs uses sync

**Source files**:
- `packages/core/hooks/scout-block.cjs` -- sync `fs.readFileSync(0)`
- `packages/core/hooks/privacy-block.cjs` -- async `process.stdin` iterator

**Issue**: Both hooks run on the same PreToolUse event in sequence. The async stdin read in privacy-block.cjs can race with pipe closure when the first hook (scout-block) already consumed stdin synchronously. In practice, JSON is piped to each hook separately by Claude Code, but the async approach adds unnecessary complexity and a 5-second timeout risk if stdin doesn't close cleanly.

**Impact**: Potential intermittent failures or slowness. Not confirmed as a current error but a design inconsistency.

**Fix**: Standardize `packages/core/hooks/privacy-block.cjs` to use `fs.readFileSync(0, 'utf-8')` like scout-block.cjs does.
