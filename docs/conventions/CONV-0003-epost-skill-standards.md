# CONV-0003: ePost Skill Authoring Standards

**Status**: Accepted
**Audience**: Agent, human
**Layer**: Layer 0 (epost_agent_kit) — org-wide standard, not per-repo deviation

This document records ePost-specific conventions layered ON TOP of the base CC skill specification (ARCH-0004). Anything here either extends CC spec (our additions) or deviates from it (with rationale). The CC spec is the floor; this doc is the ceiling.

---

## Convention 1: CSO — Cognitive Skill Optimization

**Rule:** The `description` field must contain trigger conditions only ("Use when…"). Never summarize what the skill does as a workflow.

**Why:** If the description summarizes workflow steps, Claude reads the summary and skips loading the skill body — the "Description Trap". The description is a routing signal, not a preview.

**How to apply:** Every new or modified skill description must pass the CSO test:
- Does it say **when** to load? → Pass
- Does it describe **what happens** when loaded? → Fail — rewrite as trigger conditions

```yaml
# Fail — summarizes workflow
description: Runs quality scoring on skills, outputs table of scores with pass/fail.

# Pass — trigger conditions
description: Use when auditing skill quality, scoring skills against CSO standards,
  or identifying which skills need improvement.
```

**Checklist:**
- [ ] Contains "Use when" or equivalent trigger language
- [ ] Mentions concrete user-facing trigger phrases
- [ ] Does NOT describe the workflow or output format
- [ ] Under 1024 characters

## Convention 2: Additional Metadata Fields

**Rule:** ePost skills use extended `metadata:` fields beyond what Anthropic suggests.

```yaml
metadata:
  keywords:
    - skill-quality
    - audit
    - scoring
  triggers:
    - /skill-audit
    - audit skills
    - score skills
  platforms:
    - all
  agent-affinity:
    - epost-code-reviewer
    - epost-fullstack-developer
```

| Field | Purpose | CC Default |
|-------|---------|------------|
| `metadata.keywords` | Search index terms for `/find-skill` relevance scoring | Not in CC spec — ePost addition |
| `metadata.triggers` | Explicit trigger phrases (slash commands + natural language) | Not in CC spec — ePost addition |
| `metadata.platforms` | Which platforms this skill applies to: `web`, `ios`, `android`, `backend`, `all` | Not in CC spec — ePost addition |
| `metadata.agent-affinity` | Which agents should proactively use this skill | Not in CC spec — ePost addition |

**Why:** The CC spec uses `description` as the sole routing signal. We extend with structured metadata to support `/find-skill` quality scoring and relevance ranking without relying solely on Claude's judgment.

## Convention 3: Quality Scoring — 10-Point System

**Rule:** All skills must score ≥8/10 before being considered production-ready.

| Dimension | Points | Pass Condition |
|-----------|--------|---------------|
| **Trigger format** | 2 | Description contains "Use when" or equivalent trigger language (CSO-compliant) |
| **Keywords & triggers** | 2 | `metadata.keywords` array is non-empty AND `metadata.triggers` array is non-empty |
| **Maturity indicators** | 2 | Has `references/` directory OR has `evals/` directory (or both) |
| **Completeness** | 2 | Both `name` and `description` fields are present and non-empty |
| **Invocability & length** | 2 | Has `user-invocable` field set (not null) AND SKILL.md body is substantive (>100 chars) |

**Scoring automated by:** `node .claude/skills/find-skill/scripts/lib/score-quality.cjs`

**Threshold:** Target ≥8/10. Skills below 7/10 are flagged for remediation.

## Convention 4: SKILL.md Size Target

**Rule:** Target ~200 lines for SKILL.md body. Hard cap: move content to `references/` when body exceeds 300 lines.

**Why:** CC spec says "under 5,000 words" — but this is for Claude.ai distribution. For CC agent context efficiency, shorter is better. Excessive SKILL.md size degrades all agents that have this skill loaded. Progressive disclosure means `references/` content is loaded on-demand; SKILL.md body is loaded whenever the skill is active.

**How to apply:**
- Core instructions → SKILL.md body
- Deep documentation, examples, checklists → `references/` with links from SKILL.md
- Troubleshooting tables, field schemas → `references/` not inline

## Convention 5: Skills Are in `packages/` — Not `.claude/`

**Rule:** ALL skill edits go in `packages/`. Never edit `.claude/skills/` directly.

**Why:** `.claude/` is generated output — wiped and regenerated by `epost-kit init`. Edits made directly to `.claude/` are lost on the next init. This is the #1 most expensive mistake in this project.

**After editing:** Run `epost-kit init --full --source .` to regenerate `.claude/` from `packages/`.

**Package mapping:**
| Skills under | Package |
|---|---|
| `packages/core/skills/` | Core (loaded by all agents) |
| `packages/a11y/skills/` | A11y package |
| `packages/platform-web/skills/` | Web platform |
| `packages/platform-ios/skills/` | iOS platform |
| `packages/platform-android/skills/` | Android platform |
| `packages/platform-backend/skills/` | Backend platform |
| `packages/kit/skills/` | Kit authoring tools |
| `packages/design-system/skills/` | Design system |
| `packages/domains/skills/` | Business domains |

## Convention 6: Skill Activation Patterns

**Rule:** Use `skills:` in agent frontmatter for guaranteed load. Do not rely on description-based auto-detection for critical skills.

| Activation Method | Reliability | When to Use |
|---|---|---|
| `skills: [skill-name]` in agent frontmatter | Guaranteed — full content injected at spawn | Core skills every agent needs |
| `user-invocable: true` — slash command | Explicit — user triggers manually | Workflow skills users invoke directly |
| `context: fork` + `agent:` — fork to sub-agent | Explicit dispatch | Task-oriented skills needing isolated context |
| Description-based auto-detection | Unreliable — depends on Claude's judgment | Supplementary skills, not critical paths |

**Why:** The description-based model invocation path ("model-invoked via description") sounds good in theory but scores poorly in practice. For skills that must run, wire them explicitly.

**How to apply:** New behavioral guardrail skills (verification, receiving-code-review, etc.) should be listed in the `skills:` frontmatter of every agent that needs them — not left to auto-discovery.

## Convention 7: Validation Before Claiming Done

**Rule:** After creating or modifying any skill, verify the skill index is up to date before claiming the work is done.

```bash
node .claude/scripts/generate-skill-index.cjs
```

**Why:** The skill index (`packages/core/skills/skill-index.json`) is used by agents for discovery. A stale index means agents won't find or route to new skills correctly.

**Also run:** `epost-kit init --full --source .` to propagate changes from `packages/` to `.claude/`.

## Convention 8: YAML Frontmatter — Use Inline Arrays

**Rule:** Prefer inline array syntax `[a, b, c]` over block sequence syntax for metadata fields.

```yaml
# Preferred
metadata:
  keywords: [skill-audit, quality, scoring]
  triggers: [/audit-skills, audit skills]
  platforms: [web, ios]

# Also supported (fixed parser — block sequences work)
metadata:
  keywords:
    - skill-audit
    - quality
    - scoring
```

**Why:** The `scan-skills.cjs` YAML parser was rewritten (Apr 2026) to handle both formats, but inline arrays are more readable and less error-prone. Block sequences silently returned `[]` in the previous parser — always test after using block format.

## Convention 9: The `agent-affinity` Field

**Rule:** Use `metadata.agent-affinity` to declare which agents should proactively use a skill. This is a hint, not automatic injection.

```yaml
metadata:
  agent-affinity:
    - epost-fullstack-developer
    - epost-debugger
```

**Why:** The CC spec has no concept of agent affinity. We add it to:
1. Help `find-skill` score relevance for the current agent context
2. Signal to humans which agents should have this skill in their `skills:` list
3. Enable future tooling to auto-suggest wiring

**Not automatic:** `agent-affinity` does NOT cause the skill to load. To guarantee load, add to the agent's `skills:` frontmatter list.

---

## Convention 10: Eval Coverage

**Rule:** Skills with `user-invocable: true` or critical workflow skills should have an `evals/` directory with at least a `triggering-tests.md` file.

```
skill-name/
├── SKILL.md
├── references/
└── evals/
    └── triggering-tests.md   ← at minimum
```

**Why:** Without eval tests, there's no systematic way to detect regression in triggering behavior after description changes.

## Quick Reference — New Skill Checklist

Before creating a new skill, verify:

```
[ ] Folder: kebab-case, under packages/{package}/skills/
[ ] SKILL.md: exact spelling, case-sensitive
[ ] name: matches folder name, kebab-case
[ ] description: WHAT + WHEN trigger conditions (CSO-compliant, <1024 chars)
[ ] user-invocable: set (true/false)
[ ] metadata.keywords: non-empty array
[ ] metadata.triggers: non-empty array (includes slash command if user-invocable)
[ ] metadata.platforms: set (web, ios, android, backend, or all)
[ ] No version: at root level (put under metadata.version)
[ ] No XML tags < > in frontmatter
[ ] SKILL.md body: substantive instructions (>100 chars)
[ ] After writing: run epost-kit init --full --source .
[ ] After init: run node .claude/scripts/generate-skill-index.cjs
[ ] Quality score: ≥8/10 (run find-skill scoring to verify)
```
