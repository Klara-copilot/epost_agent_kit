## Project-Specific Terminology

### Core Concepts

**Agent**
An AI assistant specialized in a specific domain. Agents receive instructions, execute tasks, and report results. In epost_agent_kit, there are 15 agents organized into global, platform-specific, and specialized categories.

**Global Agent**
One of 9 orchestration-focused agents (orchestrator, architect, implementer, reviewer, researcher, debugger, tester, documenter, git-manager) that coordinate workflows across all platforms.

**Platform Agent**
Specialized agents for web, iOS, or Android development. Execute platform-specific code and delegate back to global agents. Examples: `web/implementer`, `ios/developer`, `android/developer`.

**Specialized Agent**
Domain-specific agents outside the standard orchestration flow. Examples: `scout` (codebase search), `brainstormer` (problem-solving), `database-admin` (database optimization).

**Skill**
A reusable playbook or capability that agents activate based on context. Skills are organized by platform (core, web, iOS, Android) and category (planning, debugging, code review, etc.). 17 skills total in the kit.

**Command**
User-facing instruction that triggers agent execution. Commands use `/` prefix (e.g., `/cook`, `/test`, `/review`). 30 commands total organized into 8 categories.

**Hook**
Lifecycle event handler that executes at specific moments (SessionStart, SubagentStart, ToolCall, Stop). Used for configuration, security, and notifications. 6 primary hooks in the system.

## Architecture Patterns

**Parent-Child Delegation**
Hierarchical execution model where global agents (parents) orchestrate and delegate to platform agents (children). Enables clean separation of concerns and maintainability.

```
User → Orchestrator (parent) → Global Agent → Platform Agent (child) → Execute
```

**File Ownership Model**
Each development phase owns an exclusive set of files. Prevents race conditions in parallel development and ensures clear responsibility boundaries.

**YAML Frontmatter**
Structured metadata at top of agent/skill/command files defining properties like name, description, model, tools, color. Example:

```yaml
---
name: epost-orchestrator
description: Task routing and project management
model: haiku
color: green
tools: [Read, Glob, Grep, Bash, TaskCreate]
---
```

**Mode Composition**
Technique used in Figma design system to avoid combinatorial explosion. Instead of creating separate variables for each combination, variables compose orthogonally (e.g., 2 platforms × 7 sizes × 7 states = 1,059 variables instead of 4,802).

## Platform-Related Terms

**Claude Code**
AI-enabled code editor and IDE where agents are natively supported. Primary platform for epost_agent_kit development. Uses `.claude/` directory structure.

**Cursor**
Code editor with AI capabilities supporting Copilot-like features. Supported platform for agent distribution. Uses AGENTS.md and `.cursor/` directory.

**GitHub Copilot**
AI pair programmer integrated into VS Code. Supported platform for agent distribution. Uses `.github/` directory for agent configuration.

**MCP (Model Context Protocol)**
Standard protocol for connecting AI models to external systems. epost_agent_kit uses MCPs for:
- **XcodeBuildMCP**: iOS development automation
- **Context7 MCP**: Documentation lookups

**Xcode**
Apple's integrated development environment for iOS/macOS development. Automated via XcodeBuildMCP for compilation, testing, and simulator management.

## Workflow & Process Terms

**Plan**
Detailed implementation document created by architect agent. Contains YAML frontmatter, research findings, file ownership, implementation steps, and success criteria. Stored in `plans/YYMMDD-HHMM-{slug}/`.

**YYMMDD-HHMM Format**
Date and time format for unique identification. Example: `260205-2250` = 2026-02-05 22:50. Used in plan naming and file organization.

**Phase**
One sequential step in implementation with specific deliverables. Example: `Phase 1: Web Implementation`. Plans are divided into phases with phase dependencies.

**File Ownership**
List of files that belong to a specific phase. Prevents parallel development conflicts. Example: "Phase 1 owns: src/components/Button.tsx, src/styles/button.css"

**Block/Blockedby**
Dependencies between tasks/phases. "Phase B blocks Phase C" means Phase C cannot start until Phase B completes.

**Report**
Summary document generated by agents after task completion. Contains findings, metrics, recommendations. Stored in `plans/reports/`.

**Frontmatter**
Metadata block at top of markdown files in YAML format. Used for configuration, tracking, and automation. Example:

```yaml
---
status: in_progress
priority: P1
effort: 4h
branch: feat/auth
---
```

## Security & Access Control

**Scout Block**
Security mechanism that prevents access to heavy directories (node_modules, .git, dist, etc.) to avoid context overflow. Uses gitignore-spec pattern matching.

**Privacy Block**
Security mechanism that prevents accessing sensitive files (.env, .key, credentials.json) without explicit user approval via `APPROVED:` prefix.

**Approval Prefix**
User consent mechanism for accessing sensitive files. Format: `APPROVED:.env` allows reading .env file after user approval.

**Build Allowlist**
Whitelist of commands permitted despite scout blocking. Examples: `npm build`, `cargo build`, `docker build`. These bypass normal scout blocking.

**Secrets Detection**
Pre-commit scanning for API keys, tokens, credentials. Prevents accidental credential leakage to git repository.

## Code Quality Terms

**LOC (Lines of Code)**
Measure of code size. Standards per component:
- Agents: Max 200 LOC
- Skills: Max 300 LOC
- Commands: Max 150 LOC
- Docs: Max 800 LOC

**Code Coverage**
Percentage of code paths executed by tests. Target: 80% minimum overall, 90%+ for critical paths.

**Type Safety**
Using TypeScript strict mode to catch errors at compile time. Reduces runtime bugs.

**Linting**
Automated code style checking via ESLint. Enforced before commit.

**Conventional Commit**
Standardized commit message format: `type(scope): description`. Types: feat, fix, docs, style, refactor, test, chore, perf.

## Agent Roles

**Orchestrator**
Global router that analyzes requests, detects platform context, and delegates to specialized agents. Acts as project manager.

**Architect**
Designs solutions, creates implementation plans, and researches best practices. Spawns researcher agents in parallel.

**Implementer**
Executes implementation based on plans. Delegates to platform agents for platform-specific code.

**Reviewer**
Performs code review, security audit (OWASP), performance analysis, and task completion verification.

**Tester**
Orchestrates testing across platforms. Runs unit, integration, and E2E tests.

**Researcher**
Multi-source technology research. Answers "how should we implement X?" questions.

**Debugger**
Root cause analysis using 6-step systematic debugging framework.

**Documenter**
Maintains documentation, generates content, ensures accuracy.

**Git-Manager**
Git operations, conventional commits, pull request creation, secret scanning.

**Scout**
Fast codebase search using Glob, Grep, and context-aware prioritization.

**Brainstormer**
Problem-solving and design evaluation. Presents alternative approaches with pros/cons.

**Database-Admin**
Database design, schema optimization, query performance analysis.

## Development Workflow Terms

**Branch**
Git branch for feature/fix development. Format: `type/short-description` (e.g., `feat/user-auth`, `fix/login-bug`).

**Issue**
GitHub issue tracking feature request or bug. Referenced in commits and PRs.

**Pull Request (PR)**
Proposal to merge feature branch into main. Includes code review, automated checks, and testing.

**Merge Conflict**
Git conflict when two branches modify same lines. Resolved manually then re-tested.

**Rollback**
Reverting changes to previous known-good state if something breaks.

**`.env`**

**Hook**

## Measurement & Metrics

**Token**
Unit of text consumed by AI models. 1 token ≈ 4 characters. Used to measure context size and API costs.

**Effort Estimate**
Predicted time to complete task. Example: "2 hours" or "8 hours".

**Complexity Level**
Estimate of difficulty. Example: "simple", "medium", "complex".

**Priority**
Urgency ranking. Examples: P1 (critical), P2 (important), P3 (nice to have).

**Status**
Current state of task. Examples: pending, in_progress, completed, cancelled.

## Version Control Terms

**Main/Master**
Primary branch containing production-ready code. Protected against direct commits.

**Feature Branch**
Temporary branch for developing new feature. Merged to main via PR.

**Commit**
Snapshot of changes with message. Should follow conventional commit format.

**Push**
Upload local commits to remote repository.

**Pull**
Download remote changes to local repository.

**Rebase**
Replay commits on top of different base branch. Cleaner history than merge.

**Cherry-pick**
Apply specific commit from one branch to another.

**Optimization**
Improving performance via code changes, indexing, caching, etc.

**Profiling**

**Bottleneck**
